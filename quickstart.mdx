---
title: "Get started with examplenv"
description: "Annotate your .env files with !SECRET or !MASK-ON directives, run examplenv gen, and get safe committable example.env files in minutes."
---

examplenv turns your existing `.env` files into committable examples with a single command. You annotate the lines that contain secrets, run `examplenv gen`, and the tool writes out safe `example.env` files alongside your originals. This guide walks you through the full workflow from annotation to output.

<Steps>
  <Step title="Annotate your .env files with directives">
    Open each `.env` file and mark the lines that hold sensitive values. Add `# !SECRET` as an inline comment on any line whose value should be redacted.

    ```bash .env.dev
    STAGE=DEV
    # It cut your secret out too. By putting !SECRET in the comment
    OPENAI_API_KEY=sk-ldfkjslfjldsjflsd # !SECRET
    APP_SECRET=ThisIsASecret # !SECRET
    ```

    Non-secret values like `STAGE=DEV` are left unmarked and will be copied verbatim into the example file.
  </Step>
  <Step title="Run examplenv gen">
    From your project root, run the `gen` command. examplenv searches the current directory recursively for all `.env` files and generates a corresponding example file for each one.

    ```bash
    uv run examplenv gen
    ```

    The command prints each file pair it processes:

    ```
    Searching for .env files in: /workspaces/examplenv

    .env.dev -> example.env.dev
    .env.prod -> example.env.prod
    .env -> example.env
    ```
  </Step>
  <Step title="Review the generated files">
    Inspect the output files to confirm that secrets have been replaced and non-sensitive values have been preserved.

    ```bash example.env.dev
    # Example environment variables for .env.dev
    STAGE=DEV
    # It cut your secret out too. By putting !SECRET in the comment
    OPENAI_API_KEY=<YOUR_OPENAI_API_KEY> # !SECRET
    APP_SECRET=<YOUR_APP_SECRET> # !SECRET
    ```

    Each secret value is replaced with a `<YOUR_KEY_NAME>` placeholder derived from the variable name, making it clear to teammates exactly what they need to supply.
  </Step>
</Steps>

<Note>
  The generated `example.env.*` files contain no real secrets and are safe to commit to version control. Add them to your repository so teammates have an accurate, always-up-to-date reference for required environment variables.
</Note>
